# План рефакторинга `cock-monitor` (master-документ для ИИ-агента)

Этот документ — **основной контракт** для поэтапного рефакторинга.  
Его цель: дать агенту весь контекст, чёткий порядок работ, критерии готовности и правила, чтобы выполнять этапы **последовательно, один за другим**, без потери поведения и операционной надёжности.

Breaking changes **разрешены**, если:

1. они явно описаны в отчёте этапа;
2. обновлены `README.md`, `DEPLOY.md`, `config.example.env`, `systemd/*` (если затронуто);
3. добавлены шаги миграции/отката.

---

## 1) Продукт и границы задачи

`cock-monitor` — набор инструментов для Linux VPS:

- мониторинг `nf_conntrack` и опционально `conntrack -S`;
- алерты в Telegram по расписанию (`systemd timer` / cron), без постоянного демона;
- хранение метрик в SQLite (`METRICS_DB`);
- дополнительные сценарии: daily chart, VLESS-report, MTProxy monitor/report, incident sampler, CPU-aware shaper.

### Что важно сохранить

- Прозрачный деплой на сервере (`/opt/cock-monitor`, `/etc/cock-monitor.env`, systemd timers).
- Предсказуемое поведение алертов (cooldown/escalation/threshold semantics).
- Совместимость с low-resource VPS.

### Что можно менять радикально

- Разбиение модулей и API внутри проекта.
- Точки входа (`bin/*.sh` → `python -m ...`) при сохранении операционного UX.
- Формат внутренних таблиц SQLite через миграции.

## 2) Актуальная архитектурная картина (на момент старта)

### Уже сделано хорошо

- Есть `pyproject.toml`, базовый CI и unit-тесты.
- Есть доменная логика `conntrack` в Python (`cock_monitor/domain/conntrack_policy.py`).
- Есть репозиторий и версионирование схемы для `conntrack_samples` / `host_samples`.
- `mtproxy_module/core.py` уже фасад, а не монолит.
- Для `/chart` и `/vless_delta` в `telegram_bot` уже используются сервисы, а не subprocess.

### Ключевые проблемные зоны

1. **Остаточный orchestration-monolith в `bin/check-conntrack.sh`**  
   Скрипт остаётся перегруженным (state, env, I/O, Telegram, вызовы policy/storage, LA-alert).
2. **Shell↔Python контракт через `eval` и shell-декларации**  
   Повышает хрупкость и стоимость изменений.
3. **Нечёткие transactional границы в части SQLite (особенно MTProxy)**  
   Много мелких `commit`, нет единой атомарной операции сценария.
4. **Сохранённые subprocess-зависимости в Telegram для `/status`**  
   `status_provider` продолжает вызывать shell script.
5. **`sys.path`-инъекции в CLI-скриптах**  
   Ухудшают переносимость, тестируемость, модель упаковки.
6. **Документация плана должна быть синхронизирована с текущим кодом**  
   Старые пункты “ещё не сделано” больше неактуальны и сбивают фокус.

## 3) Целевая архитектура

Цель: устойчивые границы и тестируемые сценарии.

```text
cock_monitor/
  domain/         # чистая логика (пороги, cooldown, агрегирование)
  adapters/       # OS/HTTP/sqlite/telegram/system tools
  storage/        # репозитории + migrations
  services/       # сценарии (use-cases)
  cli/            # thin CLI wrappers (или __main__ subcommands)

telegram_bot/
  handlers + runtime, без shell-посредников для продуктовых команд

mtproxy_module/
  collector / alerts / repository / reports / charts
  (или перенос в cock_monitor/services + adapters в одном стиле)
```

### Архитектурные инварианты

- `domain` не импортирует `subprocess`, `sqlite3`, network-библиотеки.
- Все внешние I/O через `adapters`/`storage`.
- Один entrypoint = один use-case.
- Все schema-change только через явные миграции.
- Любой сложный сценарий покрыт тестами хотя бы на уровне “pure logic + repository”.

---

## 4) Правила выполнения для ИИ-агента

1. Выполнять **строго один этап за итерацию**, без смешивания целей.
2. Перед началом этапа сверять:
   - scope этапа,
   - критерии готовности,
   - список обязательных регресс-проверок.
3. После этапа обязательно обновлять docs/ops-артефакты при изменении контрактов.
4. Не вносить “архитектуру ради архитектуры”; каждое изменение должно уменьшать сложность или риск.
5. Любой breaking change сопровождать:
   - миграционными шагами,
   - обратной совместимостью или явным отказом от неё,
   - заметкой в отчёте этапа.

## 5) Порядок этапов (фиксированный)

## Этап 0 — Базовая инвентаризация и контракты

**Цель:** зафиксировать поведение до крупных изменений.

**Сделать:**

- Матрица сценариев: `entrypoint -> входы -> выходы -> side effects -> timer`.
- Перечень таблиц/колонок `METRICS_DB` и кто ими владеет.
- Перечень операционных зависимостей (утилиты, права, пути, systemd units).

**Готово, если:**

- Матрица и контракты оформлены в `docs/stage-0-inventory-and-contracts.md`.
- Есть список “критичных пользовательских сценариев” для регресса.

---

## Этап 1 — Упрощение conntrack orchestration (главный приоритет)

**Цель:** вывести логику из `bin/check-conntrack.sh` в Python use-case.

**Сделать:**

- Создать Python use-case `run_conntrack_check(...)` (или аналог).
- Перенести orchestration-логику из shell:
  - чтение state,
  - принятие решений,
  - формирование сообщений,
  - запись в storage,
  - вызов Telegram-адаптера.
- Убрать `eval` контракт shell↔python.
- `bin/check-conntrack.sh` оставить как тонкую оболочку или заменить на `python -m ...`.

**Готово, если:**

- В shell не остаётся бизнес-логики.
- Все доменные решения происходят в Python.
- Поведение `--dry-run` и текущие алерт-семантики сохранены (или явно обновлены как breaking).

---

## Этап 2 — Единая модель транзакций и storage-границ

**Цель:** атомарность сценариев и предсказуемое хранение.

**Сделать:**

- Для ключевых сценариев определить transaction boundaries.
- В `mtproxy_module/repository.py` сократить количество “внутренних” commit; сделать операции композиционными.
- Явно разделить:
  - mutate-операции,
  - read/query-операции,
  - schema/migrations.

**Готово, если:**

- Один сценарий = одна логическая транзакция (где это уместно).
- Нет неожиданных partial-write при штатных ошибках.
- Тесты проверяют happy path и rollback/failure path.

---

## Этап 3 — CLI/entrypoints без `sys.path`-хака

**Цель:** воспроизводимый запуск и чистая модель импортов.

**Сделать:**

- Убрать модификации `sys.path` из `bin/*.py`.
- Перевести entrypoints на пакетный запуск (`python -m ...`) или корректные console scripts.
- Унифицировать аргументы/ошибки/коды возврата CLI.

**Готово, если:**

- Скрипты запускаются без ручной правки `PYTHONPATH` (кроме явно документированных ops-кейсов).
- Локальный запуск и systemd запуск используют одинаковые entrypoints/контракты.

---

## Этап 4 — Telegram слой: убрать остаточные shell-зависимости

**Цель:** `telegram_bot` использует Python services/API напрямую.

**Сделать:**

- Перевести `/status` на Python service (без вызова shell-скрипта).
- Унифицировать обработку ошибок/таймаутов.
- Проверить, что команды бота не имеют скрытых side effects вне ожидаемого.

**Готово, если:**

- В `telegram_bot/status_provider.py` нет subprocess-вызовов продуктовых сценариев.
- Команды `/status`, `/chart`, `/vless_delta`, `/mt_*` стабильно работают в текущем деплое.

---

## Этап 5 — VLESS/Incident/Shaper: доведение до единых границ

**Цель:** одинаковые архитектурные принципы во всех подсистемах.

**Сделать:**

- VLESS: отделить use-case, adapters (x-ui sqlite, access log), formatting.
- Incident sampler: минимизировать дублирование с host metrics, выделить общие helpers.
- Shaper: либо оставить shell осознанно (и документировать почему), либо вынести управляющую логику в Python.

**Готово, если:**

- У каждого сценария есть “owner”-модуль и понятный flow.
- Уменьшено количество кросс-импортов и дублирования.

---

## Этап 6 — Финализация контракта и операционной модели

**Цель:** закрепить результат как стабильную платформу для дальнейших изменений.

**Сделать:**

- Полностью синхронизировать docs (`README`, `DEPLOY`, `config.example.env`, `systemd/*`).
- Обновить preflight по новым зависимостям/entrypoints.
- Ввести короткий “upgrade guide” для уже работающих инсталляций.

**Готово, если:**

- Новый разработчик может поднять проект только по документации.
- Для breaking-изменений есть понятный путь миграции.

---

## 6) Единый набор регресс-проверок после каждого этапа

Минимум:

1. `pytest`
2. `ruff check .`
3. Ручные сценарии:
   - `check-conntrack` в dry-run;
   - `python -m telegram_bot --poll-once ...` (или эквивалент нового entrypoint);
   - `/status` и `/chart` (если доступен env/токены);
   - MTProxy/VLESS smoke run (dry-run, где возможно).

Если этап затрагивает storage/migrations:

- тест миграции с пустой БД;
- тест миграции со “старой” БД-фикстурой;
- проверка сохранности данных после миграции.

---

## 7) Риски и контрольные точки

| Риск | Как снижать |
|------|-------------|
| Регресс алертов | golden tests на policy + dry-run сравнение текстов |
| Потеря/порча данных в SQLite | явные миграции, backup-step в upgrade guide |
| Ломка деплоя на сервере | сохранить совместимые systemd примеры и пути |
| Избыточная абстракция | ограничивать scope этапа и измерять упрощение |
